1. Product Overview

Golfin Garage ERP is a custom-built enterprise resource planning system for a golf cart operations, service, and build execution business. It replaces the legacy Shopmonkey platform with a purpose-built system designed around the company's specific domain: work orders, inventory management, build-slot planning, ticketing/rework workflows, technician task tracking, and QuickBooks accounting integration.

The system follows an MVP-first strategy — shipping a focused, production-ready feature set quickly, with explicit extension points planned for Phase 2. The architecture is serverless-first on AWS to minimize idle cost while preserving scale headroom.

The system is decomposed into 10 bounded contexts. Each owns its schema, routes, and event contracts.

Migration Intake Shopmonkey data import, transformation, validation POST /migration/shopmonkey/start migration.batch_imported, migration.record_failed

Platform Controls Audit logging, observability, operational alerts Internal only audit.recorded, observability.alert_triggered
Cross-context rule: Only the owning context writes to its schema. Cross-context communication is event-first — no direct cross-schema writes. Read models are generated from events or read-only SQL views.

6. MVP Feature Requirements MVP

6.1 Work Orders & Ticket Lifecycle

  • Create, list, and transition work orders through defined states
  • Ticket creation with assignment to technicians
  • Status transitions with audit trail and event emission
  • Rework/escalation subsystem for quality issues
  • QC gate checkpoints within work order workflows
  • Time entry tracking per technician per work order

6.2 Inventory Management

  • Parts catalog with lot tracking
  • Stock level monitoring with reservation mechanics
  • Shortage detection with automatic event notification
  • Vendor management and procurement tracking
  • Failure-case handling (over-reservation, concurrent modifications)

6.3 Build-Slot Planning

  • Deterministic slot-planning engine with baseline scoring algorithm
  • Capacity-aware scheduling (reject over-capacity assignments)
  • Snapshot-based plan computation for reproducibility
  • Plan publication with downstream event propagation

6.4 Customer & Dealer Management

  • Customer records with contact information
  • Dealer relationship tracking
  • Customer-linked work orders and invoices

6.5 Accounting Integration (QuickBooks)

  • Bidirectional sync: customers, invoices, payments, items, vendors, GL entries
  • Outbox pattern for reliable sync delivery
  • Reconciliation and error recovery workflows
  • Invoice sync state tracking

6.6 SOP & On-the-Job Training

  • Standard operating procedure document management
  • OJT assignment creation and step-by-step tracking
  • Training progress visibility for leads

6.7 Employee Web Dashboard

  • Role-based dashboards (admin, shop manager, technician, etc.)
  • Navigation: work orders, inventory, customers, planning, accounting, training, reporting, admin
  • Toast notifications, role context provider
  • Responsive layout via AppShell component

6.8 Floor Technician App

  • Touch-first mobile-responsive interface
  • Work order viewing and task completion
  • Shift management (start/end shift)
  • Offline-capable data sync
  • Minimal chrome via TechShell wrapper

6.9 Shopmonkey Migration

  • Import master/open records first (active customers, open tickets, parts)
  • ETL pipeline: parse → transform → deduplicate → validate → load
  • Record-level error tracking with reject queue
  • Batch checkpoints for resumable reruns
  • Audit evidence (actor, source hashes, accepted/rejected counts)

6.10 AI Capabilities (Bedrock)

  • SOP/OJT assistant (RAG over training documents)
  • Ticket summarization
  • Planning copilot
  • Inventory anomaly detection
  • Natural-language search across domain data
  • Per-feature budget controls and tool allow-lists

6.11 Platform & Observability

  • Immutable audit log pipeline with correlation IDs
  • Structured logging, CloudWatch metrics, X-Ray tracing
  • App-layer row-level authorization (fail-closed, deterministic denial codes)
  • Audit points: authn.success/failure, authz.allow/deny/scope_deny

Data Access Patterns

  • Repository pattern for contexts with complex queries: Inventory, Tickets, Accounting, Migration (e.g., work-order.repository.ts, customer.repository.ts)
  • Direct Prisma access for simple workflows: SOP/OJT, AI
  • Row-level access: filterRowsByScope() and requireScope() guards at context boundaries

Database Workflow

Schema lives at packages/db/prisma/schema.prisma. After editing: run npm run db:migrate to create a migration, then npm run db:generate to update the Prisma client. Both are enforced in CI.

9. Event-Driven Architecture

Event Contract

  • Delivery guarantee: at-least-once (consumers must be idempotent)
  • Naming: snake_case with dots — e.g., work_order.created, inventory.lot.changed
  • Versioning: in envelope/payload, not in event names
  • Atomic writes: domain write + audit write + outbox append in a single database transaction
  • Correlation: every mutation carries a correlationId; retry-safe commands include Idempotency-Key

Sync vs. Async Flow Patterns

PatternExampleType
Create ticketPOST /tickets → DB write → returnSync
Reserve inventoryPOST /inventory/reservationsSync
Compute draft planPOST /planning/slots/computeSync
Shortage check after ticketTicket created → inventory shortage checkAsync
Planning constraint updatePart reserved → recalc planning constraintsAsync
Accounting syncStatus changed → QuickBooks sync triggerAsync
Workspace refreshPlan published → dashboard projections refreshAsync

Failure Handling

Failure ModeMitigation

Planner timeoutStep Functions timeout + audit log

Missing snapshotsFail explicitly — never silently continue

10. Authentication & Authorization

  • AWS Cognito User Pool with password policy and MFA support
  • App Clients per application (web, API, floor-tech)
  • Cognito Groups map to ERP roles
  • JWT token claims: standard (sub, iss, aud, exp) + custom (custom:org_id, custom:shop_id)
  • Implementation: packages/auth/src/jwt.ts — normalizeCognitoJwtClaims()

12. Shopmonkey Migration

Strategy

  1. Master/open records first — active customers, open tickets, active parts
  2. Run ERP in production for new activity alongside legacy cutover
  3. Historical backfill in controlled batches with throttling and checkpoints

Migration Phases

PhaseDescription
1. Discovery & MappingField mapping, data quality rules, reject conditions
2. Staging ImportLoad raw Shopmonkey records into migration schema staging tables
3. Canonical TransformMap staging data → context-owned schemas via transformers
4. Cutover & ReplayFreeze legacy, reconcile deltas, emit audit evidence
5. Historical BackfillReplay historical records with throttling and checkpoints

ETL Pipeline (@gg-erp/migration)

The migration package provides a modular pipeline:

  • src/parsers/ — Parse Shopmonkey export formats
  • src/transformers/ — Map to ERP domain schema
  • src/dedup/ — Deduplication logic
  • src/validation/ — Data quality validation
  • src/loaders/ — Load into PostgreSQL
  • src/cli/ — CLI tools: fetch-shopmonkey, sanitize-shopmonkey, load-shopmonkey

14. Application Surface Area

API (apps/api)

  • Runtime: Node.js HTTP server locally, Lambda handlers in production
  • Entry points: src/server.ts (local dev), src/index.ts (Lambda)
  • Structure: contexts/ (10 domain modules), lambda/ (12 handler modules), middleware/ (6 modules), events/, audit/, observability/
  • Dependencies: @prisma/client, @aws-sdk/client-s3, @aws-sdk/s3-request-presigner
  • Lambda domains: work-orders, customers, inventory, tasks, invoices, QC gates, time entries, and more

Web Dashboard (apps/web)

  • Framework: Next.js (App Router) + React 18
  • Routes: accounting, admin, auth, customer-dealers, customers, inventory, planning, reporting, training, work-orders
  • State: Role context provider (RoleProvider), feature-specific modules in features/
  • UI: Tailwind CSS, @gg-erp/ui shared components, Sonner toasts, Lucide icons
  • Auth: aws-amplify (Cognito client-side integration)

Floor Tech (apps/floor-tech)

  • Framework: Next.js 14 + React 18 — touch-first mobile-responsive
  • Routes: work-orders, auth, shift management, data sync
  • Shell: TechShell wrapper with minimal chrome for mobile use
  • UI: Tailwind CSS, @gg-erp/ui, Lucide icons

Workers (apps/workers)

  • Runtime: Custom WorkerRuntime class with event registration and retry logic
  • Registered handlers: work_order.created, audit.event.recorded
  • Features: configurable max retries, structured logging, metrics emission
  • Pattern: runtime.register(eventName, handler) → runtime.handle(event)
